# 虚拟用户压力测试（Virtual User Stress Test）

用一群"虚拟但真实溯源"的用户，在上市前对你的方案做一轮压力测试——挑刺、找漏洞、发现极端风险。

## 何时使用

- 新品 / 新功能概念想先做一轮"会被哪些人反对、为什么"的预演
- 社媒文案 / 种草内容想预判哪些用户会反感、会反驳
- 价格策略想测用户的价格敏感度与接受边界
- 已有真实 VOC（用户评价 / 社媒反馈）想活化成可运算的虚拟用户群

## 核心原则（铁律）

1. **真实锚定**：每位虚拟用户的核心声音必须来自真实 VOC 源头，可溯源，不可凭空捏造。
2. **边界透明**：能测 = 概念/功能/文案/价格（可从标签提取的）；不可虚拟化 = 真人体验（手感/流畅度/情绪流）；前置真实反馈输入 → 可测（虚拟人复用其洞察做放大）。
3. **分层标注**：锚定层（真实）与生成层（算法推断，带置信度）必须区分，明确"哪些是生成的"。

## 方法论：四层引擎

| 层 | 名称 | 理论依据 | 纯规则近似 |
|----|------|----------|-----------|
| L1 | 场景分解 | 联合分析（Conjoint）| 正则 + 关键词词典提取属性（价格/卖点/品类）|
| L2 | 用户画像深化 | Kahneman 参考依赖 + 保护动机理论 | 结构化字段 + 关键词索引预计算（参考价/痛点词/需求标签/强度）|
| L3 | 核心匹配引擎 ★ | ELM 精细可能性模型 | 关键词重叠评分：相关性 = 场景属性 ∩ 用户标签；价格差公式 |
| L4 | 响应生成 | 叙事一致性 + 认知失调 | 模板增强（注入最相关原话 + 场景上下文）|

### 匹配引擎核心公式（纯规则实现）

```
相关性 relevance = |用户关注维度 ∩ 场景卖点维度| / |用户关注维度|
立场得分 oppose = 星级基线
              + Σ(卖点契合其痛点 ? −0.16 : 卖点契合其需求 ? −0.10 : 未触及 ? +0.05)
              + 价格偏差(参考价对比)
              + 未解决痛点余量
              + 情绪基调(负+0.10 / 正−0.05)
立场 = oppose≥0.52→反对 ; ≤0.40→支持 ; 否则中立
置信度 = clamp(|oppose−0.46|×2.2, 0.4, 0.98)
风险 = (立场=反对) ? 影响力×(0.5+强度) : 0
```

**关键逻辑**：若用户痛点是"噪音大"，场景卖点是"静音"——维度命中且痛点被解决，该用户**转为支持**，而非被错判为反对。这正是"壳子没有内核"的实现缺失点。

### 关注维度（对齐 voc-report 九大标签模块）

价格 / 静音 / 材质 / 耐用 / 清洗 / 尺寸 / 外观隐私 / 功能 / 售后 / 舒适。
VOC 标签（产品问题标签→使用阻碍、需求维度标签→需求/改进、用户属性→画像、使用场景→场景）映射到这些维度做交集计算。

## VOC 数据要求

- 源头：Amazon 评价、社媒反馈、客服记录等真实文本
- 标签体系（推荐对齐 voc-report）：用户画像 / 购买动机 / 购买目的 / 使用场景 / 使用阻碍 / 克服方式 / 用户忠诚度 / 产品改进建议 / 十三种需求
- **打不上标签很正常**：很多人没提到有价值内容，引擎仅使用有标签/有价格的样本做相关性计算

## 输出形态：双页结构

1. **用户清单**：真的有 N 张（默认 100）独立身份卡，每张可溯源到真实 VOC 原话，支持按分群/立场/极端筛选与搜索；压力测试后每张卡显示立场徽章、驱动因子与相关度。
2. **数据看板**：聚合视图——立场分布、分群×立场矩阵、反对原因聚类、高风险用户排名。两页共享同一批运算结果，可切换对照。

## 部署网页工具（GitHub Pages）

网页文件：`index.html` + `voc-source.js`（VOC 数据集）+ `favicon.svg`。
部署方式：GitHub Contents API（建仓 → PUT 文件 → 开 Pages）。部署脚本参考 `references/build_voc.py`（VOC 构建）与下方部署片段。

纯前端、零后端、无需大模型即可运行（LLM 仅用于增强语义匹配与原创叙事，非必需）。
